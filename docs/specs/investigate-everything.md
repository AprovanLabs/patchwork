@patchwork/docs/specs/unified-event-system.md contains my incoherente ramblings on how to to build a system that let's LLMs integrate with 3rd party APIs and local file systems in a generic way. I'm trying to design a system that is very generic and reminiscient of how Linux handles a wide vareity of challenges by treating everything as a text stream. I'm trying to marry all of these concepts:

- Handling incoming events via Webhooks. HTTP requests, CRON
- Handling calling 3rd party libraries and services in a unified way. Stitchery starts down this path, converting MCP/HTTP into simple namespaced function calls (RPC). However, there's a bit of a missing piece where we need a 'registry' of sorts to keep track of versioned libraries for these APIs
- Handling 'stream-based' or 'subscription' 3rd party info. MCP/HTTP are great, but how do we expose to LLMs the concept of a Websocket/SSE that they can respond to continuously? Does this go to a generic event broker? There should probably be a queue implementation with some mechanisms to batch or delay, with monitoring. How do we handle caching? Hardcopy kind of explores this
- Handling unifying 3rd party data contracts/entity links. MCP is great for input/output. How do we consistently link between the entity types in responses/params? How could we potentially link between entities in other systems? I might have links to Jira tickets in a GitHub comment. Could we rely on URLs to generate graph-like capabilities? If we hvae graph entities, we need some way to consistently store and reference the data models in this graph database. Perhaps input/output models from API endpionts are each treated in some consistent way to auto generate models?
- We need to be able to handle the different between 'events' and 'assets,' but I'm struggling to quantify this. Yes, an 'assets' could be a local file that we know values of, but could that be considered an 'event' where the body is the content and an attribute is 'current time'? How do we think of files in Git, otherwise, since assets have 'sha' as an attribute to resolve the contents. Apprentice starts to explore this, storing descriptions of versinoed pieces of data
- Handling automatic integrations with 3rd party services. Like in Patchwork/Stitchery, I'm able to add new 3rd party APIs ane immediately start generating UI widgets that rely on the data. I don't need to store any config elsewhere. Hardcopy, on the other hand, requires me to know the APIs up-front and define in-code ahead of time how I want to reference entities and queries. THIS IS A PROBLEM! Ideally, I could dynamically define interesting queries and mount paths based on whatever 3rd party APIs I've registered in the system. Then let widgets and other tools handle things
- Unifying all this with the latest 'skills' system. LLMs are centralizing on using skills, which are named directories with high-level instructions in SKILL.md files and supporting .md resources or scripts to do interesting things.
